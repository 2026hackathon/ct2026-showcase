<section class="diggy-card span-12" id="tech"><div class="section-head"><h2>Technical details</h2><p>M1 uses deterministic aggregation for a trustworthy report; M2 uses round-by-round evidence gathering for a traceable diagnosis</p></div><div class="tech-grid"><div class="tech-detail"><div class="tag">M1 · time-window report</div><h3>Turn natural language into a verifiable log report</h3><p>M1's design principle: AI handles understanding and expression, while the deterministic system handles facts and numbers. A user types "analyze the logs from the past three days" in Slack, and the system handles the time window, aggregation results, health scoring and report generation separately.</p><div class="tech-steps"><span><b>Input:</b> a natural-language Slack request; the LLM only parses the time range and intent.</span><span><b>Facts:</b> ES aggregation produces the anomaly count, top services, error codes and trends — numbers are not left to the LLM to compute.</span><span><b>Grading:</b> code computes the health score and risk grade, baking explainable rules into the flow.</span><span><b>Validation:</b> entity validation prevents fabrication — service names, error codes and numbers must trace back to the ES aggregation results.</span><span><b>Output:</b> Slack progress messages plus a Markdown / PDF report, with reviewable conclusions.</span></div></div><div class="tech-detail"><div class="tag">M2 · anchor diagnosis</div><h3>Propose a hypothesis first, then converge on the root cause with evidence</h3><p>M2 doesn't let the model guess the root cause directly — it has the model propose a hypothesis + gap, then choose the next evidence-gathering tool. Each round of evidence supports or refutes the hypothesis, and the final conclusion must be backed by citations to the raw logs.</p><div class="tech-steps"><span><b>Input:</b> a Kibana link, alert ID or traceId, to lock the diagnostic anchor first.</span><span><b>Evidence pool:</b> collects raw logs, traceId, error messages, instance lists and counter-evidence records.</span><span><b>Tools:</b> a restricted ES / AKS / mock tool layer executes by fixed signatures, with no unbounded search.</span><span><b>Convergence:</b> a hypothesis supported by evidence goes deeper, one refuted by counter-evidence switches to H2 / H3, and insufficient evidence continues to the next round.</span><span><b>Output:</b> a diagnostic report with raw-log citations and an audit trail, to aid judgment rather than replace human decisions.</span></div><div class="media tech-media"><img src="m2-flow.svg" alt="M2 round-by-round evidence-gathering Runtime Harness flow chart"></div></div></div><div class="architecture-block"><div class="caption">Core architecture</div><h3>The Slack entry routes first, then enters the Runtime Harness</h3><div class="architecture-frame"><img src="diggy-core-architecture.png" alt="Diggy core architecture diagram"></div></div></section>
<section class="diggy-card span-12"><div class="section-head"><h2>How we constrain agent behavior</h2><p>A two-layer closed loop of Coding Harness and Runtime Harness</p></div><div class="harness-grid"><div class="harness-card"><div class="tag">Coding Harness</div><h3>First converge the dev process into acceptance-ready tasks</h3><p>We don't just let AI write code freely — we use the hx flow to lock down the task, context, tests and DoD.</p><div class="harness-list"><span>hx go: pick up a task card, converge scope first</span><span>hx task start: open a worktree, isolate parallel development</span><span>hx spec / hx check: accept against the spec, always run checks when done</span><span>hx eval / hx testgen: run evals before changing a prompt, generate cases for critical modules first</span><span>hx task done: produce a DoD report and traceable commits</span></div></div><div class="harness-card"><div class="tag">Runtime Harness</div><h3>Then corral runtime behavior into four gates</h3><p>When actually handling a user request, the LLM only understands and chooses a strategy, while task state, tool calls, evidence citations and report publishing are all gated by deterministic code.</p><div class="harness-list"><span>Task Gate: create the task directory first, with clear pending / running / done</span><span>Tool Gate: every tool call must carry a hypothesis + gap</span><span>Evidence Gate: a ToolResult citation enters the pool; no match, no entry into the report</span><span>Schema + Audit Gate: pass the schema first, then publish; audit.jsonl traces the whole way</span></div></div></div><div class="harness-note">The core of these two harness layers: not making the model more obedient, but making the process not depend on the model being obedient.</div></section>
